Understanding Microservices Architecture: A Developer's Guide

R
Roshan Damor
November 15, 2024

Microservices architecture has become one of the most talked-about patterns in modern software engineering. Unlike the traditional monolithic approach where the entire application is deployed as a single unit, microservices break your application into small, independently deployable services — each responsible for a specific business capability.

What Are Microservices?

Each microservice is a small application that owns its own data store, communicates over a well-defined API (usually HTTP/REST or message queues), and can be developed, tested, and deployed independently of others. Think of it as building a city with many specialised buildings instead of one giant tower that does everything.

Key Benefits

  • Independent deployment — ship one service without touching others
  • Technology flexibility — use Node.js for one service, Python for another
  • Fault isolation — a crash in the notification service won't bring down payments
  • Easier scaling — scale only the services under load, not the entire app
  • Smaller codebases — teams own focused, understandable services

A Simple Service Definition

Here's what a minimal Express-based microservice looks like in practice:

// user-service/index.js
const express = require('express');
const app = express();

app.use(express.json());

// Owns only user-related logic
app.get('/users/:id', async (req, res) => {
  const user = await UserRepository.findById(req.params.id);
  if (!user) return res.status(404).json({ error: 'Not found' });
  res.json(user);
});

app.post('/users', async (req, res) => {
  const user = await UserRepository.create(req.body);
  // Emit event for other services (e.g. email service)
  eventBus.emit('user.created', user);
  res.status(201).json(user);
});

app.listen(3001, () => console.log('User Service on :3001'));

Common Challenges

Microservices are not a silver bullet. The distributed nature introduces problems that don't exist in a monolith: network latency, partial failures, distributed transactions, and the complexity of service discovery. You'll need to invest in observability tooling — centralised logging, distributed tracing with tools like Jaeger or Zipkin, and health checks.

Inter-Service Communication Patterns

There are two fundamental communication patterns: synchronous (direct REST or gRPC calls — good for real-time queries) and asynchronous (event-driven via Kafka, RabbitMQ, or AWS SQS — better for workflows where eventual consistency is acceptable). Always prefer async where you can — it decouples services more thoroughly and makes your system more resilient.

When Should You Use Microservices?

Microservices shine when you have multiple teams working independently, high scaling requirements for specific features, or when different parts of the system have very different tech needs. For a startup or small project, a well-structured monolith is almost always the right call. Build the monolith first, identify the seams, then extract services when the pain is real — not hypothetical.

The key is not to microservice everything from day one, but to design with clear boundaries and clean APIs so the transition, when it comes, is smooth. Start with a modular monolith and evolve from there.

Building Scalable APIs with Node.js and GraphQL Best Practices

R
Roshan Damor
October 28, 2024

REST has served us well for decades, but as frontends grew more complex and mobile came to dominate, a new problem emerged: over-fetching and under-fetching. GraphQL, originally developed at Facebook and open-sourced in 2015, solves this elegantly by letting clients ask for exactly what they need — nothing more, nothing less.

REST vs GraphQL

With REST, you create endpoints for resources: GET /users/1, GET /users/1/posts, GET /posts/5/comments. To render a user's profile page with their latest posts, you might make 3+ round trips. GraphQL collapses this into a single request where you describe the shape of data you need:

query GetProfile($userId: ID!) {
  user(id: $userId) {
    name
    avatar
    bio
    posts(limit: 5, orderBy: NEWEST) {
      title
      publishedAt
      likes
      tags { name }
    }
  }
}

Setting Up Apollo Server with Node.js

Apollo Server is the most popular GraphQL server for Node.js. Here's a production-ready setup with authentication middleware and DataLoader for N+1 prevention:

const { ApolloServer } = require('@apollo/server');
const { expressMiddleware } = require('@apollo/server/express4');
const DataLoader = require('dataloader');

const server = new ApolloServer({
  typeDefs,
  resolvers,
  plugins: [ApolloServerPluginDrainHttpServer({ httpServer })],
});

await server.start();

app.use('/graphql', expressMiddleware(server, {
  context: async ({ req }) => {
    const user = await authenticate(req.headers.authorization);
    return {
      user,
      // DataLoader batches multiple DB calls into one
      userLoader: new DataLoader(batchLoadUsers),
      postLoader: new DataLoader(batchLoadPosts),
    };
  },
}));

The N+1 Problem and DataLoader

The most common performance pitfall in GraphQL is the N+1 query problem. If you fetch a list of 20 posts and each post resolver independently queries its author, you'll fire 21 database queries. DataLoader solves this by batching all those individual IDs and making a single SELECT * FROM users WHERE id IN (...) call per request cycle.

Best Practices for Production

  • Depth limiting — prevent deeply nested malicious queries that overload your DB
  • Query complexity scoring — cap the total "cost" of a query before executing it
  • Persisted queries — only allow pre-registered queries in production for security
  • Field-level caching — use @cacheControl directives with a CDN like CloudFront
  • Schema versioning — use deprecation directives, never break a field without warning

Subscriptions for Real-Time Data

GraphQL Subscriptions via WebSockets let you push data to clients when events occur — perfect for live notifications, dashboards, or chat. Apollo Server supports this out of the box with graphql-ws. Define a subscription type in your schema, resolve it with an AsyncIterator backed by a pub/sub system (Redis Pub/Sub is the standard choice), and your clients get live updates.

GraphQL isn't a replacement for REST in every scenario — for simple CRUD APIs or public third-party integrations, REST is still the right tool. But for complex, data-driven frontends with multiple consumers, GraphQL's flexibility and strong typing make it a compelling choice.

Docker and Kubernetes: Complete Guide to Container Orchestration

R
Roshan Damor
September 12, 2024

Containers have fundamentally changed how we build, ship, and run software. Before Docker, the classic headache was "it works on my machine" — environment mismatches between dev, staging, and production caused endless bugs. Containers package your application with everything it needs: the runtime, libraries, environment variables, and config — making environments reproducible by design.

Why Containers Changed Everything

A container is not a virtual machine. VMs virtualise the hardware, running a full OS for each instance. Containers share the host OS kernel and isolate only the process and filesystem — making them orders of magnitude lighter, faster to start (milliseconds vs minutes), and much more efficient to run at scale.

Writing a Production Dockerfile

A poorly written Dockerfile creates bloated images and slow builds. Here's a multi-stage build for a Node.js app — it compiles TypeScript in a build stage and copies only the compiled output into the final lean image:

# ── Build stage ───────────────────────────
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY . .
RUN npm run build          # compiles TypeScript → /app/dist

# ── Production image ──────────────────────
FROM node:20-alpine
WORKDIR /app
ENV NODE_ENV=production

# Copy only what we need
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules

# Run as non-root for security
USER node
EXPOSE 3000
CMD ["node", "dist/index.js"]

Kubernetes Core Concepts

Kubernetes (K8s) is the de-facto standard for orchestrating containers at scale. The key abstractions you'll use daily are: Pod (one or more containers that share a network namespace), Deployment (manages desired state and rolling updates for a set of Pods), Service (stable DNS + load balancing in front of Pods), and Ingress (HTTP routing from the outside world into Services).

A Real-World Deployment Manifest

apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-server
spec:
  replicas: 3
  selector:
    matchLabels:
      app: api-server
  template:
    metadata:
      labels:
        app: api-server
    spec:
      containers:
        - name: api
          image: roshandamor/api-server:v1.4.2
          ports:
            - containerPort: 3000
          resources:
            requests: { cpu: "100m", memory: "128Mi" }
            limits:   { cpu: "500m", memory: "512Mi" }
          readinessProbe:
            httpGet: { path: /healthz, port: 3000 }
            initialDelaySeconds: 5
          env:
            - name: DATABASE_URL
              valueFrom:
                secretKeyRef:
                  name: app-secrets
                  key: database-url

Zero-Downtime Deployments

One of Kubernetes's most valuable features is rolling updates. When you push a new image, K8s gradually replaces old Pods with new ones, keeping at least a configurable number of healthy replicas available at all times. Combined with proper readinessProbe configuration, traffic is only routed to Pods that have passed their health check — so your users feel nothing during a deploy.

Observability in K8s

Running containers at scale without observability is flying blind. The standard stack is: Prometheus for metrics scraping, Grafana for dashboards, Loki for log aggregation, and Jaeger or Tempo for distributed tracing. Helm charts make standing all of this up straightforward — the kube-prometheus-stack chart gives you a full monitoring suite in a single command.

Containers and Kubernetes have become essential skills in the modern developer's toolkit. Start with Docker, get comfortable building and optimising images, then layer on Kubernetes once you understand that the ultimate goal isn't just running containers — it's building self-healing, scalable, observable systems.
